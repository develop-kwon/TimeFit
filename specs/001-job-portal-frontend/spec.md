## User Scenarios & Testing

### User Story 1 - 회원가입 (Sign Up) (Priority: P1)

구직자와 채용자는 각각의 역할에 맞는 회원가입 절차를 통해 서비스에 가입할 수 있다. 구직자는 기본 정보 입력 후 관심 직종을 선택하고, 채용자는 업체명과 기본 정보를 입력하여 가입한다.

**Why this priority**: 서비스의 가장 기본적인 진입점이며, 사용자 온보딩에 필수적이다. 성공적인 회원가입은 후속 서비스 이용의 전제 조건이다.

**Independent Test**: 구직자와 채용자 각각의 회원가입 흐름을 처음부터 끝까지 독립적으로 테스트할 수 있으며, 성공적으로 가입된 계정으로 로그인하여 각자의 메인 화면에 접근하는 것을 검증한다.

**Acceptance Scenarios**:

1.  **Given** 사용자가 회원가입 페이지에 접속했을 때, **When** '구직자' 탭을 선택하고 모든 필수 정보를 입력한 후 가입을 시도하면, **Then** 2단계로 진행되고, 관심 직종을 선택 후 가입이 완료된다.
2.  **Given** 사용자가 회원가입 페이지에 접속했을 때, **When** '채용자' 탭을 선택하고 모든 필수 정보를 입력한 후 가입을 시도하면, **Then** 가입이 즉시 완료된다.
3.  **Given** 사용자가 회원가입 페이지에 접속했을 때, **When** '데모 모드' 토글을 활성화하면, **Then** 데모 모드 관련 UI 변경이 적용된다.
4.  **Given** 사용자가 회원가입 페이지에 접속했을 때, **When** '이미 계정이 있으신가요? 로그인' 링크를 클릭하면, **Then** 로그인 페이지로 이동한다.

### User Story 2 - 구직자 일자리 찾기 및 지원 (Job Seeker Job Search & Apply) (Priority: P1)

구직자는 다양한 필터 기능을 활용하여 자신에게 맞는 일자리를 검색하고, 스케줄에 기반한 자동 매칭 결과를 확인하며, 원하는 일자리에 지원할 수 있다. 지원 후에는 지원 상태가 반영된다.

**Why this priority**: 구직자 서비스의 핵심 기능으로, 사용자가 서비스를 통해 일자리를 찾는 본질적인 목적을 달성하게 한다.

**Independent Test**: 구직자 계정으로 로그인 후 일자리 검색, 고급 필터 적용, 스케줄 매칭 방식 변경, 특정 일자리에 지원하는 전체 과정을 독립적으로 테스트할 수 있다. 지원 후 상태 변경을 검증한다.

**Acceptance Scenarios**:

1.  **Given** 구직자가 로그인 후 일자리 찾기 화면에 접속했을 때, **When** 검색창에 키워드를 입력하고 검색하면, **Then** 관련 일자리 목록이 표시된다.
2.  **Given** 구직자가 로그인 후 일자리 찾기 화면에 접속했을 때, **When** '고급 필터'를 클릭하고 카테고리, 시급 범위, 근무 형태, 날짜, 시간 필터를 적용하면, **Then** 필터링된 일자리 목록이 표시된다.
3.  **Given** 구직자가 스케줄을 등록하지 않은 상태로 일자리 찾기 화면에 접속했을 때, **When** 화면을 로드하면, **Then** "등록한 스케줄이 없습니다." 메시지와 "마이페이지로 이동" 버튼이 표시된다.
4.  **Given** 구직자가 스케줄을 등록한 상태로 일자리 찾기 화면에 접속했을 때, **When** '내 스케줄에 맞는 일자리만 보여주기' 옵션이 선택되면, **Then** 등록된 스케줄과 시간이 겹치는 일자리가 우선적으로 노출된다.
5.  **Given** 구직자가 특정 일자리 카드에서 '지원하기' 버튼을 클릭했을 때, **When** 지원 요청이 성공적으로 전송되면, **Then** 해당 버튼이 '지원 대기 중'으로 변경된다.

### User Story 3 - 구직자 마이페이지 관리 (Job Seeker My Page Management) (Priority: P2)

구직자는 마이페이지에서 자신의 스케줄을 효율적으로 관리하고, 지원한 일자리 목록을 확인하며, 개인 프로필 정보를 수정할 수 있다.

**Why this priority**: 서비스 사용의 편의성을 높이고, 구직자가 자신의 활동을 추적하고 관리할 수 있게 하여 사용자 유지에 기여한다.

**Independent Test**: 구직자 계정으로 로그인 후 마이페이지의 각 탭(스케줄, 활동, 프로필)을 이동하며 스케줄 추가/삭제, 지원 목록 확인, 프로필 정보 수정 기능을 독립적으로 테스트할 수 있다.

**Acceptance Scenarios**:

1.  **Given** 구직자가 마이페이지 '스케줄' 탭에 접속했을 때, **When** '+ 추가' 버튼을 클릭하여 스케줄 등록 모달에서 날짜, 시작/종료 시간, 근무 가능 여부를 설정하고 저장하면, **Then** 스케줄이 리스트에 추가된다.
2.  **Given** 구직자가 마이페이지 '활동' 탭에 접속했을 때, **When** 화면을 로드하면, **Then** 지원한 일자리 목록이 각 일자리의 이름, 회사명, 지원 날짜, 현재 상태와 함께 표시된다.
3.  **Given** 구직자가 마이페이지 '프로필' 탭에 접속했을 때, **When** 기본 정보를 수정하고 저장하면, **Then** 프로필 정보가 업데이트된다.

### User Story 4 - 채용자 구인글 관리 (Employer Job Posting Management) (Priority: P1)

채용자는 구인 관리 화면에서 현재 모집 중인 구인글과 매칭 완료된 구인글을 확인하고, 새로운 구인글을 작성하여 등록할 수 있다.

**Why this priority**: 채용자 서비스의 핵심 기능으로, 채용자가 인력을 모집하는 본질적인 목적을 달성하게 한다.

**Independent Test**: 채용자 계정으로 로그인 후 구인 관리 화면에서 구인글 목록 확인, 상태 탭 전환, 새로운 구인글 작성 및 등록 과정을 독립적으로 테스트할 수 있다.

**Acceptance Scenarios**:

1.  **Given** 채용자가 로그인 후 구인 관리 화면에 접속했을 때, **When** '모집 중' 탭을 클릭하면, **Then** 현재 모집 중인 구인글 목록이 표시된다.
2.  **Given** 채용자가 로그인 후 구인 관리 화면에 접속했을 때, **When** 등록된 구인글이 없으면, **Then** "등록된 구인글이 없습니다." 메시지가 표시된다.
3.  **Given** 채용자가 구인 관리 화면에서 '+ 구인글 작성' 버튼을 클릭했을 때, **When** 구인글 작성 폼에서 모든 필수 정보를 입력하고 '등록' 버튼을 클릭하면, **Then** 구인글이 저장되고 목록에 '지원자 0명'과 함께 표시된다.

### User Story 5 - 채용자 지원자 관리 (Employer Applicant Management) (Priority: P2)

채용자는 마이페이지의 '지원자 관리' 기능을 통해 등록된 구인글별로 지원자 목록을 확인하고, 상세 화면에서 지원자를 선택하거나 인터뷰를 요청하는 등의 액션을 취할 수 있다.

**Why this priority**: 채용자가 구인글에 대한 지원자들을 효과적으로 관리하고 채용 절차를 진행할 수 있도록 지원하여, 서비스의 가치를 높인다.

**Independent Test**: 채용자 계정으로 로그인 후 마이페이지에서 '지원자 관리' 탭을 통해 특정 구인글의 지원자 목록을 확인하고, 지원자를 선택/인터뷰 요청하는 과정을 독립적으로 테스트할 수 있다.

**Acceptance Scenarios**:

1.  **Given** 채용자가 마이페이지 '지원자 관리' 탭에 접속했을 때, **When** 등록된 구인글 목록을 확인하고 특정 구인글을 클릭하면, **Then** 해당 구인글에 지원한 구직자들의 목록을 볼 수 있는 상세 화면으로 이동한다.
2.  **Given** 채용자가 지원자 상세 화면에 접속했을 때, **When** 특정 지원자를 선택하거나 인터뷰 요청 버튼을 클릭하면, **Then** 해당 액션이 처리된다.

### Edge Cases

- 사용자 입력 필드 유효성 검사 실패 시 처리 (예: 이메일 형식, 필수 필드 누락).
- API 통신 오류 발생 시 사용자에게 적절한 피드백 제공.
- 네트워크 연결 끊김 상황에서의 UI/UX 처리.
- 검색 결과가 없을 때의 UI 표시.
- 스케줄 등록 시 중복 시간대 처리.
- 비활성 계정 데이터 삭제 절차 관련 처리.

## Requirements

### Functional Requirements

- **FR-001**: 시스템은 구직자와 채용자 유형을 선택할 수 있는 회원가입 UI를 제공해야 한다. (image_e76bd7.jpg, image_e763ff.jpg 참조)
- **FR-002**: 시스템은 구직자 회원가입 시 기본 정보(이름, 이메일, 비밀번호, 연락처, 주소)를 1단계로 입력받고, 2단계에서 관심 직종을 복수 선택하도록 허용해야 한다. (image_e7645a.jpg 참조)
- **FR-003**: 시스템은 채용자 회원가입 시 업체명, 이메일, 비밀번호, 연락처, 주소를 단일 단계로 입력받아야 한다. (image_e763ff.jpg 참조)
- **FR-004**: 시스템은 모든 회원가입 필수 필드에 대한 유효성 검사를 수행해야 한다.
- **FR-005**: 시스템은 회원가입 및 로그인 페이지 하단에 '데모 모드' 토글과 '이미 계정이 있으신가요? 로그인' 링크를 제공해야 한다.
- **FR-006**: 시스템은 구직자가 일자리 제목, 업체명으로 검색할 수 있는 텍스트 입력 필드를 제공해야 한다.
- **FR-007**: 시스템은 구직자 일자리 검색 시 카테고리, 시급 범위(슬라이더), 근무 형태, 날짜(Date Picker), 시간(드롭다운)을 포함하는 고급 필터 기능을 제공해야 한다. (image_e69205.png 참조)
- **FR-008**: 시스템은 구직자가 '내 스케줄에 맞는 일자리만 보여주기' 또는 '내 스케줄에 맞지 않는 모든 일자리 보여주기' 옵션을 선택하여 일자리 노출 방식을 제어할 수 있도록 해야 한다.
- **FR-009**: 시스템은 구직자가 등록한 스케줄이 없을 경우 일자리 목록 대신 "등록한 스케줄이 없습니다." 메시지와 "마이페이지로 이동" 버튼을 표시해야 한다.
- **FR-010**: 시스템은 구직자의 등록된 스케줄과 시간이 겹치는 일자리를 일자리 목록에 우선 노출해야 한다.
- **FR-011**: 시스템은 구직자가 일자리에 '지원하기' 버튼을 클릭하면 지원 요청을 처리하고, 해당 버튼의 상태를 '지원 대기 중'으로 변경해야 한다. (image_e6ef79.png 참조)
- **FR-012**: 시스템은 구직자가 마이페이지 '스케줄' 탭에서 스케줄 추가 모달을 통해 날짜, 시작/종료 시간, 근무 가능 여부를 설정하여 스케줄을 등록/관리할 수 있도록 해야 한다. (image_e6f417.png 참조, image_e6f073.png 참조)
- **FR-013**: 시스템은 구직자가 마이페이지 '활동' 탭에서 지원한 일자리의 이름, 회사명, 지원 날짜, 현재 지원 상태를 목록 형태로 확인할 수 있도록 해야 한다. (image_e6ec74.png 참조)
- **FR-014**: 시스템은 구직자가 마이페이지 '프로필' 탭에서 자신의 기본 정보를 확인하고 수정할 수 있도록 해야 한다. (image_e6ebd8.png 참조)
- **FR-015**: 시스템은 채용자 구인 관리 화면에 '모집 중', '매칭 완료' 탭을 제공하여 구인글 상태별 필터링을 가능하게 해야 한다. (image_e612fe.png 참조)
- **FR-016**: 시스템은 채용자가 등록한 구인글이 없을 경우 "등록된 구인글이 없습니다." 메시지를 표시해야 한다. (image_e60b1e.png 참조)
- **FR-017**: 시스템은 채용자가 제목, 설명, 위치, 시급, 날짜, 시작/종료 시간, 장기 근무 토글, 업종, 근무 형태를 입력하여 새로운 구인글을 작성하고 등록할 수 있도록 해야 한다. (image_e60a5e.png 참조)
- **FR-018**: 시스템은 채용자가 구인글 등록 후 해당 구인글 목록에 '지원자 0명'과 함께 표시하도록 해야 한다. (image_e612fe.png 참조)
- **FR-019**: 시스템은 채용자 마이페이지에 '지원자 관리' 탭을 제공하고, 등록된 구인글별로 지원자 목록을 확인할 수 있도록 해야 한다.
- **FR-020**: 시스템은 채용자가 지원자 상세 화면에서 지원자를 선택하거나 인터뷰를 요청하는 액션을 취할 수 있는 UI를 제공해야 한다.
- **FR-021**: 시스템은 API 통신 실패 또는 시간 초과 시 사용자에게 친절한 메시지를 표시하고 "재시도" 옵션을 제공해야 한다. (SC-002 및 Edge Cases와 연관)
- **FR-022**: 시스템은 모든 통신에 HTTPS를 사용해야 한다.
- **FR-023**: 시스템은 JWT 기반 인증을 사용하여 사용자를 인증해야 한다.
- **FR-024**: 비활성 계정의 데이터는 업계 표준 관행(예: 90일)에 따라 관리되어야 하며, 데이터 저장에는 사용자 동의가 필요하다.
- **FR-025**: 구직자의 지원 상태는 'Submitted', 'Screening', 'First Interview', 'Second Interview', 'Final Interview', 'Offered', 'Declined Offer', 'Hired', 'Withdrew'로 관리되어야 한다.
- **FR-026**: 구인글 삭제 시 관련 지원 내역은 아카이브 처리되어야 한다.
- **FR-027**: 사용자 계정 삭제 시 관련 데이터(지원 내역 포함)는 영구 삭제 전 일정 기간 아카이브 처리되어야 한다.
